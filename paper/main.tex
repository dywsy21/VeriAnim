\begin{abstract}
LLM 3D animation generation should produce more than a rendered clip: it should produce an executable scene whose objects, motions, camera views, and failures can be inspected and repaired. Blender programs provide a practical target for this goal because object construction, keyframes, cameras, materials, and rendering all have programmable interfaces. Animation, however, introduces temporal obligations that static scene generation does not expose: object identities must persist, event windows must own their transforms, contacts must hold through interpolation, cameras must show the relevant motion, and unsupported dynamics must fail with clear scope. We present VeriAnim, a verifier-facing layer for LLM-generated Blender animations built around a static \code{SceneSpec} IR, an Animation Extension Contract, and VeriAnim-AnimBench. VeriAnim first generates and repairs a static scene, then extends that measured baseline with animation events, family-specific probes, media artifacts, deterministic checks, visual/video verification, and bounded repair. VeriAnim-AnimBench contributes 300 natural-language prompts across easy, medium, and hard animation tiers for evaluating scene complexity, temporal composition, verifier coverage, and repair cost.
\end{abstract}

\section{Introduction}

Many 3D tasks become meaningful only after objects move. A robotics mockup needs the approach, contact, lift, transfer, and release of a gripper, an educational animation needs the order of a process, a game prototype needs timed interactions, and a synthetic-data scene needs controllable object and camera motion. The open source 3D modeling software, Blender, makes these requirements programmable: its Python API exposes object construction, materials, cameras, lights, keyframes, and render settings as executable program structure~\citep{blender2026pythonapi}.

Recent LLM progress makes this programmable 3D medium a realistic target for natural-language generation. Modern language models generate code, call tools, observe failures, and refine outputs across programming tasks~\citep{chen2021evaluating,austin2021program,yao2023react,schick2023toolformer,madaan2023selfrefine,shinn2023reflexion}; coding agents extend this pattern by turning a request into an artifact, running it, and using runtime feedback to repair it. The question is therefore no longer whether an LLM can write some Blender Python, but whether it can generate an animation whose motion, camera evidence, and temporal relations survive execution. A prompt such as ``a gripper moves a box from a conveyor to a cart while the camera follows'' requires the model to build the scene, schedule events, maintain contact, preserve object identity, and render evidence that the requested motion actually happened.

The main obstacle is that static 3D generation and 3D animation ask generated code to satisfy different contracts. Prior code-based 3D modeling shows that LLMs can generate static Blender programs with inspectable objects, materials, cameras, and layouts~\citep{lu2025verianim}. Static verification can inspect a finished scene: the objects exist, the relations hold, and the camera sees them. Animation verification must reason over frame windows, because success depends on how objects move between keyframes, which controller owns each transform, whether contact persists during interpolation, and whether the camera captures the relevant event. A direct keyframe script can reach the correct final pose while passing through a support surface, carrying an object without the gripper, or hiding the contact point in the rendered video.

VeriAnim addresses this gap by making generated Blender animations verifier-facing from the start. It first builds a static scene through \code{SceneSpec}, a typed contract over objects, parts, materials, relations, cameras, proxies, and evidence; the harness executes, verifies, and repairs that static baseline before adding motion. The \emph{Animation Extension Contract} then attaches animation families, event windows, capability profiles, simulation-cache metadata, and verifier probes to the measured scene. During execution, the harness records Blender traces, screenshots, preview videos, deterministic reports, and visual/video verifier feedback, and it uses that measured evidence to repair only the local placement or motion window that failed.

This paper makes four contributions:
\begin{itemize}
    \item \textbf{Static and temporal IR contracts.} We define \code{SceneSpec} for static Blender scenes and an animation extension that organizes families, probes, capability profiles, event windows, and artifact ownership.
    \item \textbf{Verifier-facing Blender harness.} We use Blender code as a 3D medium and add staged static-to-animation generation, static and temporal repair, visual/video verification, and media diagnostics.
    \item \textbf{Rigid primitive methodology.} We instantiate the contract with geometry-aware rigid primitives for support motion, paired interaction, pick-place manipulation, articulation, and temporal composition.
    \item \textbf{VeriAnim-AnimBench.} We release a 300-prompt benchmark with easy, medium, and hard tiers for measuring scene complexity, temporal composition, verifier coverage, and repair cost.
\end{itemize}
The current implementation supports static scene generation and repair, rigid and camera animation events, texture resolution, deterministic and media verifier loops, deformable Blender events, capability-profile reporting, and benchmark generation.

\section{Related Work}

\paragraph{LLMs as program generators.}
Large language models have moved from text completion toward executable work: they synthesize programs, call tools, interleave reasoning with actions, and refine outputs from feedback~\citep{chen2021evaluating,austin2021program,yao2023react,schick2023toolformer,madaan2023selfrefine,shinn2023reflexion}. Blender scripting fits this trajectory because scene construction and rendering already have stable APIs. Animation raises the requirement from syntactic execution to temporal correctness, so the generated program must expose the objects, events, probes, and artifacts that validation needs.

\paragraph{Code-based 3D modeling.}
LL3M shows that Blender programs can serve as the primary output format for large language 3D modeling~\citep{lu2025verianim}. VeriAnim uses that codebase as a reference point, but changes the problem from static modeling to verifier-facing animation. The added structure is temporal: event windows, family-specific probes, video evidence, runtime capability profiles, and repair contracts become first-class parts of generation.

\paragraph{3D modeling IRs and scene representations.}
Existing 3D representations organize different parts of the modeling problem. USD and COLLADA focus on production assets, composition, animation channels, and interchange metadata~\citep{pixar2016usd,khronos2025collada}. ShapeAssembly and StructureNet capture object part structure through programs or hierarchical graphs~\citep{jones2020shapeassembly,mo2019structurenet}, while 3D scene graphs and graph-conditioned generators encode objects as nodes and relations as edges~\citep{armeni2019scenegraph,gao2024graphdreamer}. Holodeck and SceneScript show that structured commands can support language-guided environment generation and reconstruction~\citep{yang2024holodeck,avetisyan2024scenescript}. \code{SceneSpec} uses the same representational instinct for a different contract: before animation begins, an LLM-generated Blender program must expose stable ids, collision proxies, camera evidence, verifier probes, and repair scope.

\paragraph{Text-to-3D, text-to-motion, and video generation.}
Generative 3D and video models answer a complementary question. Text-to-3D methods synthesize assets, rooms, or multiview-consistent scenes from language and image priors~\citep{poole2022dreamfusion,lin2023magic3d,hollein2023text2room,shi2024mvdream,tang2024dreamgaussian}; text-to-motion methods generate human motion trajectories~\citep{tevet2023mdm,zhang2023t2mgpt}; and text-to-video systems generate temporal media directly~\citep{ho2022imagenvideo,blattmann2023align}. These outputs can look temporally plausible, but they do not give a Blender script with editable object ids, contact windows, transform ownership, and repairable keyframes. VeriAnim targets that executable intermediate state.

\paragraph{Motion primitives and task structure.}
Robotics systems often decompose language-conditioned action into skills, affordances, and motion primitives~\citep{huang2022inner,ahn2022can}; classical planning and animation make a related move by separating paths, constraints, and motion structure~\citep{lavalle2006planning,witkin1988spacetime}. VeriAnim adopts this decomposition for generated Blender code rather than robot control. A primitive in our harness is a contract over generated keyframes and the evidence that checks them: support height, contact offset, gripper phase, hinge axis, frame window, and media visibility.

\paragraph{Physics simulation and animation plausibility.}
Physics engines and position-based methods provide collision response, constraints, particles, fluids, and soft bodies~\citep{coumans2016bullet,muller2007position,macklin2014unified}. Generated Blender scripts often use explicit transforms instead, because keyframes are easier for an LLM to write than a stable simulation setup. VeriAnim treats simulation support as a capability profile inside the extension contract. Rigid primitives, bounding-box probes, BVH checks, joint probes, particle statistics, and deformation probes all attach to the same event structure, so the harness can represent both kinematic and simulated families without pretending they have the same runtime evidence.

\paragraph{Multimodal verification for generated content.}
Vision and video-language models provide semantic checks over rendered artifacts~\citep{openai2023gpt4v,li2023blip2,liu2023llava}, while deterministic validators measure geometry and temporal predicates directly. The two evidence sources fail in different ways: a video model can judge whether an orbit shot communicates the intended subject, but a deterministic probe can catch a one-frame support violation hidden by camera framing. VeriAnim keeps both sources in the loop and records their disagreement as an evaluation target.

\section{Methodology}

VeriAnim treats animation generation as a sequence of contracts rather than a single code-generation step. The pipeline first asks an LLM to describe and generate a static scene, executes that program in Blender, and repairs bounded scene failures before any motion is introduced. Only after this baseline is measured does the animation stage add event windows, family-specific helpers, media artifacts, and verifier probes. The same run then produces transform traces, screenshots, and preview videos, allowing deterministic checks and visual/video verifiers to diagnose failures against the same object ids and frame windows. Repair operates on those measured failures, so a bad support height changes the local placement or path rather than regenerating unrelated scene content.

\subsection{Static Scene Contracts Anchor Animation}

\code{SceneSpec} makes the static scene explicit because every animation event later depends on stable object identities and measured spatial relations. The IR owns the object inventory, part structure, materials, cameras, lights, environment, collision proxies, screenshot plan, and texture policy. This separation lets the planner state semantic intent while the generated Blender program remains an executable realization of that intent. \code{ObjectSpec} records controllable entities, roles, dimensions, parts, materials, and proxies; \code{SpatialRelationSpec} records support, attachment, containment, distance, and facing constraints together with the verifier that will check them; and \code{CameraSpec}, \code{ScreenshotPlan}, and \code{MaterialSpec} specify the evidence that visual verification and texture resolution require.

The static-scene-first boundary gives repair a stable target. If a support check fails, the repair routine can move the misplaced object relative to the measured support top instead of regenerating the table, camera, and material. If a later animation check fails, the repair can edit the keyed path while preserving the verified scene. This locality matters because visual feedback, deterministic geometry checks, and LLM refinement all refer to the same frozen object ids.

\subsection{Animation Extension Contracts Make Time Explicit}

The Animation Extension Contract is the temporal layer on top of \code{SceneSpec}. It adds four kinds of information that static scene generation does not need: animation families, event windows, verifier probes, and runtime capability profiles. The contract also forces every artifact to keep the same object ids, so generated Blender code, transform traces, deterministic reports, screenshots, preview videos, and repair prompts all describe the same entities.

\textbf{Families define behavior scope.}
The current IR recognizes rigid, camera, visibility, deformable, and mixed animation families. Rigid and camera events have executable Blender paths today, while deformable events use visible shape change and deformation statistics as their initial evidence. Requests outside the implemented runtime, such as full particle or fluid behavior, become scoped capability-profile reports instead of silently producing incomplete artifacts.

\textbf{Probes define acceptance evidence.}
Each family names the evidence that can validate it. Rigid events use bounding boxes, BVH checks, or convex proxies; camera events use coverage and subject-visibility reports; deformable events combine deformation statistics with video evidence. Because these probes attach to the same event structure, the harness can compare deterministic and media judgments instead of treating them as separate review channels.

\subsection{Staging Preserves the Static Scene Baseline}

The runtime enforces the static-to-animation boundary by measuring the scene before any temporal code runs. After executing the static script, the Blender addon validates ids and relations, applies deterministic support repair for bounded placement errors, renders screenshots, and runs visual verification. The animation stage then references the verified ids when it adds keyframes, media sampling, contact constraints, or camera events. This ordering prevents an animation repair from hiding a scene-construction error by rebuilding the scene.

Blender serves as the shared measurement substrate for every later verifier. The addon records object summaries, aggregates descendants by \code{verianim\_id}, samples transforms at semantic frames and dense audit windows, renders screenshots, and renders preview videos when requested. These artifacts give deterministic validators, video verifiers, and LLM refiners a common factual record of the generated script.

Each primitive is a verifier-facing helper with preconditions over roles, generated keyframes, geometric invariants, and observable predicates. The taxonomy follows the structure of animation intent rather than a catalog of implementation failures. Natural-language prompts usually describe relations such as on, inside, holding, attached, following, and opening, and each relation implies a different invariant. Support motion owns vertical placement, carry motion owns a driver-payload offset, gripper manipulation owns a phase sequence, hinged motion owns an origin and axis, and composition owns transform ownership across time windows. Development traces helped calibrate tolerances and checks, but the primitive families come from these invariants.

\subsection{Verification Combines Geometry and Media}

Deterministic verification checks geometric predicates over sampled frames. Support predicates require footprint overlap and a small vertical gap, nonpenetration predicates measure overlap depth across collision-enabled pairs, carry-contact predicates distinguish intended proximity from embedding, and containment predicates check whether centers or bounding boxes remain inside a container proxy. Dense audit windows matter because a start/middle/end sample can miss an interpolation path that briefly passes through another object.

Media verification answers the semantic and framing questions that geometry cannot answer alone. Vision checks judge static layout, material intent, relation visibility, and camera framing; video checks judge temporal order, apparent motion, final state, and subject visibility. VeriAnim sends video artifacts, ordered sampled frames, transform traces, and deterministic reports together, so multimodal feedback can point to the same object ids and frame windows as the deterministic evidence.

\subsection{Repair Uses Measured Evidence}

Repair uses the measured failure record to choose the smallest edit that can address the violation. Static support repair recalculates object placement from the support top before animation begins; temporal support repair recalculates the subject's vertical coordinate at keyed frames; path repair shifts keyed locations only in the failing window; and bridge/platform repair constructs outside-lift-enter-cross-exit paths when an object must cross a support boundary. Carry repair sends role-specific evidence to the refiner when a script violates gripper or payload ownership. The loop stops on repeated identical failure signatures, which prevents refinement from running indefinitely without new evidence.

\section{System Components}

The implemented harness consists of a Blender addon server, Python runner, typed IR parser, texture resolver, visual verifier, video verifier, deterministic validator, and refinement loop. The addon exposes code execution, scene inspection, material inspection, camera coverage reports, screenshot rendering, animation sampling, and preview video rendering. Each run stores the artifacts needed to reproduce and inspect the generation path: IR snapshots, generated and refined scripts, verifier reports, media paths, selected textures, transform traces, and model configuration.

Texture resolution is handled as an auditable preprocessing step rather than an implicit side effect of code generation. The planner marks materials with \code{texture\_policy}, \code{needs\_texture}, and \code{texture\_query}; a material agent searches public texture pages, downloads candidates, asks a vision model to approve surface suitability, and writes approved local paths back into the IR. When search or approval fails, the coder falls back to procedural colors and shader parameters. This keeps appearance assets visible in the artifact trail while keeping the animation claim focused on executable motion.

Prompt caching keeps refinement iterative without repeating the whole system description on every call. The stable \code{verianim\_utils} API contract appears in a fixed prompt prefix shared by coder, refiner, user-edit, and animation-extension calls, while per-run IR, verifier reports, and media evidence stay in the user payload. The result is a refinement loop whose context changes with the evidence, not with the helper-library contract.

\section{Benchmark and Evaluation}

VeriAnim-AnimBench evaluates whether the contract, primitive library, and repair loop improve executable animation generation. Executable code benchmarks show that LLM claims need task-level artifacts and pass/fail measurements~\citep{chen2021evaluating,jimenez2024swebench}; AnimBench applies that principle to Blender animation by keeping the prompt, generated code, Blender artifacts, verifier reports, and repair history together. Figure~\ref{fig:evaluation-plan} shows the main comparison: prompt families span rigid support motion, paired interaction, manipulation, articulation, camera motion, visibility/state changes, deformable events, and mixed scenes, while system variants remove one layer at a time.

The benchmark contains 300 natural-language prompts stored in \code{benchmark/}: 100 easy, 100 medium, and 100 hard. Easy prompts isolate one object or one spatial relation, medium prompts combine two coordinated events or camera requirements, and hard prompts require multi-object composition, concurrent motion, or mixtures across animation families. Each record stores the prompt, animation families, required motions, verifier focus, and difficulty rationale, so benchmark runs can be analyzed by both prompt complexity and verifier type.

\textbf{Showcase audit.}
Before running the full benchmark, the curated \code{showcase/} set acts as a qualitative audit of implemented behavior. These examples do not replace benchmark-scale statistics, but they exercise the same artifact trail: source prompts, generated Blender code, scene images, animation previews, validation reports, and exported \code{.blend} files. The current showcase covers static relation checks, single-support motion, repaired multi-support motion, and composed rigid events. Representative entries include table-top sliding and conveyor motion with deterministic, vision, and video checks passing; road-ramp-platform and table-ramp-platform examples where contact repair turns an initial deterministic failure into a passing animation; and composed scenes such as garage-door opening followed by car motion, marble-to-cup transfer, and cube collision with final validation reports passing.

\textbf{Metrics.}
The primary metrics measure both correctness and refinement cost: deterministic pass rate, support violation rate, penetration rate, final relation success, transform-ownership violations, subject visibility, video verifier pass rate, number of refinement rounds, and wall-clock execution cost. Deformable events add deformation-statistic pass rate, while capability-profile cases report missing joint, particle, or volume probes as unsupported scope.

\textbf{Ablations.}
The ablations remove one layer at a time to isolate the source of each gain. Removing the extension contract tests whether a flat animation spec gives the LLM enough structure; removing primitives tests whether model-authored keyframes satisfy the same event contracts; removing dense-frame audits tests whether sparse samples miss interpolation errors; and removing video verification tests whether deterministic geometry alone captures temporal intent and camera coverage.

\textbf{Independent audits.}
The benchmark also includes independent audit hooks because several rigid primitives and deterministic checks share bounding-box assumptions. Human ratings judge visual plausibility and semantic success, mesh-level BVH checks audit rigid scenes, and deformation statistics validate shape change beyond video impressions. Disagreement among deterministic, media, and human judgments becomes a reported result for verifier design rather than an error to hide.

\section{Limitations}

VeriAnim provides a contract layer and a runnable Blender harness, but it does not replace a full physics simulator. The current rigid primitives are kinematic: they preserve support, contact, offsets, and frame-window ownership, but they do not model forces, friction, mass, torque, or collision response. Bounding boxes over-approximate concave objects and miss fine mesh intersections, so mesh-level audits remain important for rigid scenes. Deformable events currently use deformation statistics and video evidence in place of a full cloth solver, and character, mocap, IK, fluid, smoke, and particle requests become capability-profile reports when runtime probes are unavailable. Video verification helps judge temporal intent and visibility, while deterministic probes remain necessary for short contact violations and hidden geometry errors.

\section{Conclusion}

LLM-generated 3D animation requires more than asking a model to write keyframes. VeriAnim turns the task into staged program synthesis: first establish a verified static scene, then add temporal events whose objects, frame windows, probes, media artifacts, and repair scope are explicit. The Animation Extension Contract makes this structure portable across animation families, while the current rigid-body primitives show how support, carry, pick-place, articulation, and composition can become geometry-aware Blender helpers. The result is an auditable path from natural-language animation requests to executable Blender programs, measured evidence, and bounded repair.
